# Инструкция по безопасной передаче инфраструктуры проекта

В этом документе описан оптимальный и безопасный процесс передачи управления тремя ключевыми компонентами инфраструктуры (GitHub, Netlify, Cloudflare) на аккаунты партнёра.

## Общая стратегия

Передача инфраструктуры — это критический процесс, который может привести к даунтайму (недоступности сайта), если выполнить его в неправильном порядке. 

**Золотое правило:** Сначала передаётся код (GitHub), затем хостинг (Netlify), и только в самом конце — управление доменом (Cloudflare).

---

## Шаг 1. Передача репозитория GitHub

GitHub позволяет легко передать права владения репозиторием другому пользователю или организации. При этом сохраняется вся история коммитов, ветки и настройки.

### Процесс передачи:
1. Партнёр должен создать аккаунт на GitHub (если его ещё нет).
2. В вашем аккаунте зайдите в репозиторий `landing-factory`.
3. Перейдите в **Settings** -> **General**.
4. Прокрутите в самый низ до раздела **Danger Zone**.
5. Нажмите **Transfer ownership**.
6. Введите имя пользователя (username) партнёра и подтвердите действие.
7. Партнёр получит email со ссылкой для подтверждения. Он должен принять приглашение в течение 24 часов.

### Что произойдёт:
- Репозиторий переместится в аккаунт партнёра.
- Старые ссылки на репозиторий будут автоматически перенаправляться на новый адрес.
- **Важно:** Вы автоматически станете соавтором (collaborator) в переданном репозитории, поэтому не потеряете доступ сразу.

---

## Шаг 2. Передача сайта в Netlify

Netlify связывает сайт с конкретным репозиторием GitHub. Поскольку репозиторий сменил владельца, Netlify нужно перенастроить, но безопаснее всего передать сам сайт внутри Netlify.

### Оптимальный процесс (без даунтайма):
1. Партнёр создаёт аккаунт в Netlify.
2. Вы приглашаете партнёра в свою команду (Team) в Netlify как **Owner** (Владелец).
   - Идите в **Team settings** -> **Members** -> **Add member**.
3. Партнёр принимает приглашение.
4. Теперь партнёр может перенести сайт в свою собственную команду:
   - В настройках сайта: **Site configuration** -> **General** -> **Site information** -> **Transfer site**.
   - Выбирает свою личную команду как цель.
5. После успешного переноса партнёр должен переподключить репозиторий GitHub (так как он теперь в его аккаунте):
   - **Site configuration** -> **Build & deploy** -> **Continuous Deployment** -> **Repository** -> **Link repository**.

### Альтернативный процесс (через поддержку):
Если у вас бесплатный тариф и вы не можете добавить партнёра как Owner, вам придётся обратиться в поддержку Netlify:
1. Соберите данные: имя сайта (`devpartners-staging`), привязанный домен (`devpartners.ru`) и email аккаунта партнёра в Netlify.
2. Перейдите на [netlify.com/support](https://netlify.com/support) и выберите опцию переноса (transfer).

---

## Шаг 3. Передача домена в Cloudflare

Это самый деликатный этап. Cloudflare не позволяет просто "передать" доменную зону другому пользователю одной кнопкой. Домен нужно заново добавить в новый аккаунт.

### Процесс передачи (минимизация даунтайма):
1. Партнёр создаёт аккаунт в Cloudflare.
2. В **своём** аккаунте Cloudflare экспортируйте текущие DNS-записи:
   - Выберите домен `devpartners.ru` -> **DNS** -> **Records**.
   - Нажмите **Export** (скачается файл зоны).
3. Партнёр в **своём** аккаунте нажимает **Add a Site** и вводит `devpartners.ru`.
4. Cloudflare предложит тарифный план (партнёр выбирает Free).
5. Cloudflare просканирует DNS. Партнёр должен импортировать файл зоны, который вы ему передали, чтобы убедиться, что все записи (особенно A-запись на Netlify `75.2.60.5`) на месте.
6. Cloudflare выдаст **новые NS-серверы** (nameservers) для аккаунта партнёра.
7. **Критический шаг:** Вы (или партнёр) идёте к регистратору домена (там, где домен был куплен, например, Reg.ru или Ru-Center) и меняете старые NS-серверы Cloudflare на новые NS-серверы, выданные партнёру.

### Что произойдёт:
- Обновление NS-серверов может занять от нескольких минут до 24 часов.
- В течение этого времени трафик будет идти либо через ваш Cloudflare, либо через Cloudflare партнёра. Поскольку DNS-записи идентичны, **даунтайма не будет**.
- Как только новые NS-серверы пропишутся, домен в аккаунте партнёра станет **Active**, а в вашем — **Moved Away** (и через 7 дней удалится).

---

## Резюме рисков и рекомендации

1. **Порядок действий критичен:** GitHub -> Netlify -> Cloudflare.
2. **SSL-сертификаты:** При переносе Netlify-сайта SSL-сертификаты сохраняются. Однако при смене Cloudflare-аккаунта убедитесь, что в Netlify домен всё ещё имеет статус "Primary domain" и SSL активен.
3. **Доступ Manus:** Если партнёр будет использовать Manus, ему потребуется сгенерировать новый GitHub Personal Access Token в своём аккаунте и добавить его в настройки проекта Manus, так как ваш токен перестанет работать для этого репозитория после отзыва ваших прав.
