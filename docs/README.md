# Documentation System

Этот каталог — единый слой долгосрочной документации проекта **landing-factory**. Его задача — хранить не временные заметки, а устойчивое знание: стандарты, процессы, архитектурные решения и проектные пакеты по каждому лендингу.

## Принцип

> **GitHub является единственным источником правды для кода, проектной документации, принятых решений и истории смысловых изменений.**

Любой артефакт, который нужен дольше одной рабочей сессии, должен либо попасть в эту структуру, либо быть осознанно отброшен.

## Структура

| Раздел | Назначение |
|---|---|
| `standards/` | Правила именования, хранения контента, ассетов и работы с Git |
| `playbooks/` | Повторяемые операционные сценарии: обновление вординга, деплой, rollback, operating model |
| `adr/` | Architecture Decision Records — фиксируют системные решения проекта |
| `projects/<slug>/` | Полный пакет знаний по конкретному лендингу |

## Правило масштабирования

Каждый новый лендинг получает **единый slug**, который используется во всех слоях проекта:

| Слой | Путь |
|---|---|
| Код страницы | `src/pages/<slug>.astro` |
| Документация | `docs/projects/<slug>/` |
| Ассеты | `public/assets/<slug>/` |

## Что не должно храниться здесь

В этот каталог не попадают временные заметки, сырые выгрузки из чатов, промежуточные scratch-файлы и артефакты без долгосрочной ценности. Репозиторий должен оставаться операционной системой проекта, а не архивом мыслительного шума.

## Операционный контур

Для сквозных задач, где исполнитель проходит путь от аудита до релиза, используйте `playbooks/agent-operating-model.md`. Этот документ связывает standards, playbooks и project-docs в единый рабочий контур и помогает не путать временные внешние инструкции с источником правды внутри репозитория.
