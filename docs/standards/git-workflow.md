# Git Workflow Standard
Этот стандарт фиксирует минимальный git-процесс для фабрики лендингов, где важны маленькие проверяемые изменения и прозрачная история.
## Базовый принцип
Один архитектурный шаг должен оформляться как один логический commit с одним проверяемым результатом. Это правило снижает риск смешивания контекстов и делает rollback предсказуемым.
## Рабочий цикл
| Шаг | Правило |
|---|---|
| Подготовка | Перед изменением нужно понимать, какой именно пакет результата создаётся |
| Внесение правок | Не смешивать в одном пакете код, governance и случайные побочные правки без необходимости |
| Проверка | Каждый пакет должен быть локально проверяемым по содержимому и git-статусу |
| Commit | Сообщение коммита должно описывать один смысловой результат |
| Push / deploy | Выполнять только после того, как зафиксирован понятный и проверенный commit |
## Правило чистоты staging-изменений
Если в рабочем дереве есть несвязанные изменения, в commit должен попадать только целевой набор файлов. Остальные изменения нужно либо оставить вне индекса, либо оформить отдельным пакетом.
## Связь с rollback
Чем чище и атомарнее коммиты, тем безопаснее откат. Поэтому качество git-workflow напрямую влияет на устойчивость deploy и rollback-процесса.
