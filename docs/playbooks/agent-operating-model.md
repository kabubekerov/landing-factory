# Agent Operating Model

Этот playbook переводит общие принципы агентной работы в операционный формат, совместимый с текущей архитектурой **landing-factory**. Его цель — не заменить существующие стандарты, а связать их в последовательный рабочий контур, который помогает вносить изменения в репозиторий без хаоса, ложных предположений и смешивания временных артефактов с долгосрочным знанием.

## Назначение

Документ нужен для ситуаций, где агент или исполнитель не просто меняет один текст, а ведёт полный цикл работы: изучает контекст, определяет источник правды, подготавливает пакет изменений, проверяет его, выпускает результат и, при необходимости, откатывает его.

## Иерархия источников правды

При конфликте между артефактами приоритет должен определяться явно, а не по памяти исполнителя.

| Уровень | Источник | Как использовать |
|---|---|---|
| 1 | Фактическое состояние репозитория | Проверять существующие маршруты, assets, docs и процессы до начала правок |
| 2 | ADR и standards в `docs/` | Использовать как системные правила, если они не противоречат коду и актуальной структуре |
| 3 | Playbooks в `docs/playbooks/` | Применять как повторяемые сценарии исполнения |
| 4 | Проектные документы `docs/projects/<slug>/` | Использовать для конкретного лендинга или серии правок |
| 5 | Временные внешние инструкции | Консолидировать в системные документы, не использовать как постоянный источник правды сами по себе |

> Если внешний документ описывает полезный процесс, он должен быть перенесён в структуру `docs/` в устойчивой форме. Сам по себе внешний markdown-файл не считается частью operating model, пока не встроен в репозиторий.

## Режимы работы

Перед началом любой задачи исполнитель должен явно определить режим работы.

| Режим | Когда применять | Что является результатом |
|---|---|---|
| Audit | Когда нужно понять текущее состояние репозитория, структуры или деплойного контура | Аудит, карта рисков, список точек вмешательства |
| Specification | Когда нужно перевести замысел в конкретный шаблон, data pack, visual brief или implementation packet | Устойчивый проектный документ в `docs/projects/<slug>/` |
| Implementation | Когда уже понятны входы и нужно менять код, assets или конфигурацию | Атомарный набор файлов с проверяемым результатом |
| Release | Когда изменения готовы к проверке build, commit, push и deploy | Зафиксированный выпускной пакет и проверка результата |
| Recovery | Когда deploy сломан, staging деградировал или надо быстро откатить рискованную правку | Откат, rollback-решение или фикс с минимальным blast radius |

## Минимальный task contract

Перед началом implementation или release-работы должен быть собран минимальный task contract. Детальные правила вынесены в `docs/standards/task-contract.md`, а здесь зафиксирован только общий принцип. Для повседневной работы можно сразу использовать `docs/templates/task-contract-template.md`, чтобы не собирать contract с нуля.

| Поле | Что должно быть определено |
|---|---|
| Scope | Какие файлы и слои проекта допускается менять |
| Source of truth | На какие документы и файлы опирается работа |
| Deliverable | Какой артефакт должен появиться в репозитории |
| Validation | Чем будет проверяться результат |
| Exclusions | Что явно нельзя менять в рамках текущего пакета |

Если хотя бы одно из этих полей не определено, задача считается неготовой к безопасной реализации.

## Release gates

Ни один пакет изменений не должен считаться готовым только потому, что код написан или документ создан. До выпуска нужно пройти следующие ворота качества. Детальный выпускной порядок вынесен в `docs/playbooks/release-gates.md`. Если нужен готовый выпускной каркас для project-doc, используйте `docs/templates/project-release-packet-template.md`.

| Gate | Проверка |
|---|---|
| Scope gate | В diff попали только целевые файлы |
| Structure gate | Изменения соответствуют текущей архитектуре репозитория |
| Content gate | В результат не просочились сырые заметки, временные заглушки или text dump |
| Validation gate | Есть локальная или структурная проверка результата |
| Git gate | Commit остаётся атомарным и описывает один смысловой результат |

## Правило анти-дрейфа

Нельзя по умолчанию переносить паттерн из одного слоя проекта в другой только потому, что он похож. Например, наличие markdown-driven routes не означает, что все новые лендинги должны собираться только через markdown. Перед масштабированием паттерна нужно проверить реальный эталонный маршрут, структуру страницы, asset-политику и действующий release-контур.

## Recovery protocol

Если во время реализации обнаружен сбой, нужно сначала ограничить риск, а уже затем исправлять всё остальное. Детальная последовательность восстановления вынесена в `docs/playbooks/recovery-protocol.md`. Для фиксации инцидента и границ аварийного пакета используйте `docs/templates/incident-report-template.md`.

| Ситуация | Действие |
|---|---|
| Непонятный конфликт между шаблонами | Остановить масштабирование, вернуться к эталонной странице и зафиксировать расхождение в docs |
| Build error | Исправлять минимальный набор файлов, не смешивая фикс со смысловыми доработками |
| Staging показывает неожиданный результат | Проверить route priority, assets и фактический deploy, не делать выводы только по памяти |
| Сомнение в правильности репозитория или ветки | Остановить push до подтверждения источника правды |

## Что считается хорошим результатом

Хороший результат в **landing-factory** — это не просто рабочая страница. Это такой пакет изменений, который:

1. укладывается в текущую архитектуру;
2. оставляет понятный след в docs;
3. может быть проверен другим исполнителем без обращения к истории чата;
4. при необходимости безопасно откатывается.
