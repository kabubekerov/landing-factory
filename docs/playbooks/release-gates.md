# Release Gates Playbook

Этот playbook фиксирует минимальные ворота качества перед commit, push и deploy в **landing-factory**. Его задача — остановить выпуск в точке, где изменения уже выглядят почти готовыми, но ещё не доказали свою совместимость с текущей архитектурой, контентным качеством и release-контуром.

## Назначение

Release gates применяются к любому пакету изменений, который влияет на код страницы, маршруты, assets, routing aliases, staging или проектную документацию, связанную с выпуском.

## Набор ворот

Перед выпуском должны быть пройдены все следующие проверки.

| Gate | Что проверить | Когда считать пройденным |
|---|---|---|
| `scope_gate` | В diff попали только целевые файлы | Несвязанные изменения не попали в commit |
| `structure_gate` | Изменения укладываются в существующую архитектуру репозитория | Не нарушены naming rules, slug structure и docs-layout |
| `content_gate` | В релиз не просочились text dump, временные заглушки, черновые заметки и placeholders | Страница и docs выглядят как устойчивый артефакт, а не рабочий черновик |
| `asset_gate` | Все обязательные assets существуют и лежат в ожидаемых путях | Отсутствуют битые ссылки и пустые слоты |
| `validation_gate` | Есть локальная структурная или функциональная проверка | Build, route check, visual smoke review или equivalent выполнены |
| `git_gate` | Commit остаётся атомарным и описывает один смысловой результат | История понятна и пригодна для безопасного rollback |

## Порядок прохождения

Release gates полезны только тогда, когда проходят в логическом порядке. Сначала нужно проверить scope и structure, затем content и assets, и только после этого выполнять validation и выпускной git-шаг.

## Связь с task contract

Release gates не должны применяться в отрыве от task contract. Перед выпуском исполнитель должен убедиться, что текущий diff всё ещё соответствует `docs/standards/task-contract.md`: не расширился scope, не изменился source of truth и не исчезла проверка результата.

## Минимальный release checklist

Для коротких задач допускается использовать сокращённый checklist.

| Проверка | Да / Нет |
|---|---|
| Менялись только целевые файлы? |  |
| Нет сырого текстового мусора и placeholders? |  |
| Все нужные assets существуют по ожидаемым путям? |  |
| Есть хотя бы одна явная проверка результата? |  |
| Commit атомарный и не смешивает несколько смыслов? |  |

## Stop rule

Если хотя бы один gate не пройден, пакет не должен идти в push или deploy только потому, что работа заняла много времени. В этом случае исполнитель либо доделывает пакет, либо понижает его до статуса черновика и явно фиксирует блокеры.

## Практическое применение

Если нужен готовый шаблон выпускного пакета, используйте `docs/templates/project-release-packet-template.md`. Он помогает собрать scope, validation, gate review и recovery note в одном документе внутри `docs/projects/<slug>/`.
