# Recovery Protocol Playbook

Этот playbook фиксирует минимальный порядок действий при сбоях в **landing-factory**. Его задача — не дать исполнителю смешать аварийный фикс, расширение scope и смысловые улучшения в один хаотичный пакет, особенно когда staging уже деградировал или release дал неожиданный результат.

## Назначение

Recovery protocol применяется в ситуациях, где нужно ограничить риск и восстановить предсказуемое состояние проекта: при build errors, routing-конфликтах, битых assets, деградации staging, ошибке выбора репозитория или сомнении в источнике правды.

## Базовый принцип

> Сначала ограничить blast radius, затем восстановить работоспособность, и только после этого возвращаться к улучшениям.

## Последовательность действий

| Шаг | Что делать |
|---|---|
| `1. stop expansion` | Остановить масштабирование и не добавлять новые улучшения в тот же пакет |
| `2. verify source of truth` | Перепроверить репозиторий, ветку, эталонную страницу, docs и фактическое состояние файлов |
| `3. isolate failure` | Локализовать минимальный набор файлов и причин, который объясняет сбой |
| `4. apply minimal fix` | Исправить только то, что нужно для восстановления ожидаемого состояния |
| `5. validate recovery` | Выполнить build, route check, visual smoke review или equivalent, чтобы подтвердить восстановление |
| `6. document residue` | Если после фикса остаются хвосты, явно описать их как follow-up, а не маскировать под finished state |

## Типовые сценарии

| Ситуация | Правильное действие |
|---|---|
| `build_error` | Не смешивать build fix с copy refresh, visual redesign или governance cleanup |
| `staging_mismatch` | Проверить route priority, alias rules, фактический deploy и assets, а не делать вывод по предположению |
| `template_conflict` | Вернуться к эталонной странице и документировать расхождение до масштабирования паттерна |
| `wrong_repo_or_branch` | Остановить push до подтверждения источника правды |
| `content_dump_in_release` | Снять релизный статус с пакета, локализовать источник dump и удалить его минимальным фикс-коммитом |

## Чего делать нельзя

Во время recovery нельзя по умолчанию расширять задачу до «раз уж всё равно трогаем, давайте заодно улучшим ещё и это». Такое поведение увеличивает blast radius, усложняет rollback и делает диагностику ненадёжной.

## Связь с task contract и release

Recovery не отменяет task contract, а временно сужает его. Перед фиксом нужно заново определить минимальный scope аварийного вмешательства и явно перечислить, что запрещено расширять. Если инцидент связан с выпуском, recovery должен завершаться повторным прохождением релевантных release gates.

## Recovery outcome

Хороший recovery-результат — это не обязательно идеальная страница или окончательно красивый релиз. Это восстановленное, объяснимое и проверяемое состояние, после которого follow-up задачи можно делать уже в нормальном implementation-режиме.

## Практическое применение
